
```text
1. B 矩阵输入改为 CSR 格式
2. A/B 最大维度 512
3. 每个 PE 的 B Buffer 保存完整 B 矩阵，可以接受
4. 细化 Decompress Unit、Aggregation Unit、ADD 阵列、C CSR Writer
5. 保留“动态剩余目标 + 最近边界切分”的行级调度方法
```

---

# 一、整体架构目标

本加速器用于执行稀疏矩阵乘法：

[
C = A \times B
]

其中 A、B、C 均为稀疏矩阵。

对于 A 中一个非零元素：

[
A(i,k)
]

它需要与 B 的第 k 行所有非零元素相乘：

[
A(i,k) \times B(k,j)
]

所以 A 中每个非零元素的真实任务量由 B 的第 k 行非零数决定：

[
work(A(i,k)) = nnz(B[k,:])
]

如果每个 PE 内有 `N_MAC` 个乘法器，则该任务对应的周期权重为：

[
cycle(A(i,k)) =
\left\lceil
\frac{nnz(B[k,:])}{N_MAC}
\right\rceil
]

本设计的核心思想是：
**根据 B 的行非零数估计每一行 A 的真实计算量，然后通过“动态剩余目标 + 最近边界切分”把连续 A 行分配给不同 PE。**

---

# 二、整体硬件流程

完整数据流如下：

```text
Host 指令
   ↓
Fetch / Decode
   ↓
Load Module 从 DRAM 读取 A CSR 和 B CSR
   ↓
Global Buffer 缓存矩阵数据
   ↓
Scheduler 统计任务量并生成 PE 任务表
   ↓
各 PE 读取自己负责的 A 行
   ↓
每个 PE 本地保存完整 B CSR
   ↓
PE 内部执行 A(i,k) × B(k,:)
   ↓
MUL 阵列产生 partial products
   ↓
Aggregation Unit + ADD 阵列完成行内累加
   ↓
Partial Row Buffer 得到 C 当前行
   ↓
Output Buffer 暂存 C 行结果
   ↓
C CSR Writer 生成 C_row_ptr / C_col_idx / C_val
   ↓
Store Module 写回 DRAM
```

---

# 三、输入矩阵格式

## 1. A 矩阵：CSR 格式

A 使用 CSR：

```text
A_row_ptr
A_col_idx
A_val
```

其中：

```text
第 i 行 A 的非零元素范围：

p ∈ [A_row_ptr[i], A_row_ptr[i+1])

A(i,k):
    k = A_col_idx[p]
    a = A_val[p]
```

---

## 2. B 矩阵：CSR 格式

你现在希望 B 也改成 CSR，这是合理的，而且会让 Scheduler 和 PE 计算都更简单。

B CSR 为：

```text
B_row_ptr
B_col_idx
B_val
```

其中：

```text
第 k 行 B 的非零元素范围：

q ∈ [B_row_ptr[k], B_row_ptr[k+1])

B(k,j):
    j = B_col_idx[q]
    b = B_val[q]
```

这样 `A(i,k)` 需要访问 `B(k,:)` 时，可以直接通过：

```text
B_row_ptr[k] ~ B_row_ptr[k+1]
```

访问 B 的第 k 行。

---

# 四、B 改成 CSR 后，Scheduler 如何改变

之前 B 是 CSC 时，Scheduler 需要做：

```text
B CSC → b_row_nnz
B CSC → B row-wise view
```

现在 B 直接输入 CSR 后，Scheduler 明显简化。

---

## 1. 不再需要 B Row View Builder

B CSR 天然支持按行访问：

```text
B(k,:)
```

所以原来的：

```text
B Row View Builder
```

可以删除。

也不需要再做：

```text
CSC scatter 到 row-wise view
```

因为输入的 B 已经是 row-wise view。

---

## 2. B Row NNZ Counter 也可以简化

B 每一行的非零数可以直接由 `B_row_ptr` 得到：

[
b_row_nnz[k] = B_row_ptr[k+1] - B_row_ptr[k]
]

硬件实现：

```text
for k = 0 to K - 1:
    b_row_nnz[k] = B_row_ptr[k+1] - B_row_ptr[k]
```

所以原来的：

```text
B Row NNZ Counter
```

可以改成：

```text
B Row NNZ Generator
```

或者：

```text
B Row Length Generator
```

它不再需要扫描所有 `B_col_idx` 或 `B_row_idx`，只需要读取 `B_row_ptr` 并做减法。

---

## 3. Scheduler 新流程

B 改成 CSR 后，Scheduler 的流程变成：

```text
Decode 指令
   ↓
读取 A_row_ptr / A_col_idx
读取 B_row_ptr
   ↓
B Row Length Generator
计算 b_row_nnz[k] = B_row_ptr[k+1] - B_row_ptr[k]
   ↓
A Row Workload Analyzer
计算每一行 A 的 row_cyc[i]
   ↓
Remaining-Aware Row Partitioner
执行动态剩余目标 + 最近边界切分
   ↓
Task Descriptor Generator
生成每个 PE 的 row_start / row_end / A_ptr_start / A_ptr_end
```

相比 B CSC 方案，少了：

```text
1. B CSC 扫描统计行 nnz
2. B row-wise view 构造
3. CSC 到 CSR 的 scatter
```

因此硬件更简单，预处理时间也更短。

---

# 五、Scheduler 模块完整说明

Scheduler 是整个架构中负责**任务分析与任务分配**的模块。

它不参与具体乘法计算，而是在 PE 计算前完成调度。

Scheduler 内部建议包含：

```text
1. Inst Buffer
2. Decode Unit
3. B Row Length Generator
4. A Row Workload Analyzer
5. Remaining-Aware Row Partitioner
6. Task Descriptor Generator
7. PE Task Table
```

---

## 1. Inst Buffer

功能：

```text
缓存 Host 下发的指令描述符
```

指令内容包括：

```text
A_row_ptr 基地址
A_col_idx 基地址
A_val 基地址

B_row_ptr 基地址
B_col_idx 基地址
B_val 基地址

C_row_ptr 基地址
C_col_idx 基地址
C_val 基地址

M, K, N
nnz_A
nnz_B
N_PE
N_MAC
启动信号
```

---

## 2. Decode Unit

功能：

```text
解析 Inst Buffer 中的指令
生成 Load / Store / Scheduler / PE 的控制信息
```

例如：

```text
矩阵维度
矩阵基地址
CSR 数组长度
PE 数量
MAC 数量
是否启动调度
是否启动计算
```

---

## 3. B Row Length Generator

由于 B 已经是 CSR，B 每一行的 nnz 直接由 `B_row_ptr` 得到。

硬件流程：

```text
for k = 0 to K - 1:

    start = B_row_ptr[k]
    end   = B_row_ptr[k+1]

    b_row_nnz[k] = end - start
```

输出：

```text
b_row_nnz[k]
```

它用于后续计算：

```text
A(i,k) 的有效乘法数 = b_row_nnz[k]
A(i,k) 的周期权重 = ceil(b_row_nnz[k] / N_MAC)
```

由于最大维度是 512，所以：

```text
b_row_nnz SRAM 只需要 512 项
```

每项可以用 10 bit 或 16 bit 保存。

---

## 4. A Row Workload Analyzer

该模块负责计算每一行 A 的真实工作量。

对于 A 的第 i 行：

```text
A(i,k0), A(i,k1), A(i,k2), ...
```

有效乘法数量：

[
row_eff[i] =
\sum_{A(i,k)\neq0}
b_row_nnz[k]
]

周期任务量：

[
row_cyc[i] =
\sum_{A(i,k)\neq0}
\left\lceil
\frac{b_row_nnz[k]}{N_MAC}
\right\rceil
]

硬件流程：

```text
for i = 0 to M - 1:

    row_eff_acc = 0
    row_cyc_acc = 0

    a_start = A_row_ptr[i]
    a_end   = A_row_ptr[i+1]

    for p = a_start to a_end - 1:

        k = A_col_idx[p]

        b_nnz = b_row_nnz[k]

        eff = b_nnz
        cyc = ceil(b_nnz / N_MAC)

        row_eff_acc += eff
        row_cyc_acc += cyc

    row_eff[i] = row_eff_acc
    row_cyc[i] = row_cyc_acc

    TotalEffectiveWork += row_eff_acc
    TotalCycleWork     += row_cyc_acc
```

其中真正用于任务分配的是：

```text
row_cyc[i]
```

因为 PE 的执行瓶颈由周期数决定。

---

## 5. Remaining-Aware Row Partitioner

这是你的核心调度模块。

它的输入是：

```text
row_cyc[i]
TotalCycleWork
N_PE
```

输出是：

```text
每个 PE 负责的 A 行范围
```

也就是：

```text
PE0: row_start[0] ~ row_end[0]
PE1: row_start[1] ~ row_end[1]
PE2: row_start[2] ~ row_end[2]
...
```

---

### 5.1 动态剩余目标

当前已经分配了一部分任务：

```text
assigned_work
```

则剩余任务量为：

```text
remaining_work = TotalCycleWork - assigned_work
```

剩余 PE 数量为：

```text
remaining_pe = N_PE - cur_pe
```

当前 PE 的动态目标为：

[
dynamic_target =
\left\lceil
\frac{remaining_work}{remaining_pe}
\right\rceil
]

这样可以避免固定 TargetCycle 导致最后一个 PE 吃掉大量剩余行。

---

### 5.2 最近边界判断

当前 PE 已经累积：

```text
cur_load
```

当前 A 行的任务量为：

```text
w = row_cyc[i]
```

加入当前行后：

```text
new_load = cur_load + w
```

当：

```text
new_load >= dynamic_target
```

时，比较：

```text
err_before = dynamic_target - cur_load
err_after  = new_load - dynamic_target
```

如果：

```text
err_after <= err_before
```

则当前行给当前 PE。

否则当前行留给下一个 PE。

同时需要一个保护条件：

```text
如果 cur_load == 0，即当前 PE 还没有分到任何行，
则必须把当前行分给当前 PE。
```

这样可以避免超大行一直被推到最后一个 PE。

---

## 6. Task Descriptor Generator

Partitioner 得到每个 PE 的行范围后，Task Descriptor Generator 生成任务描述符。

每个 PE 的任务描述符为：

```text
PE_task_desc[pe_id] = {
    valid,
    row_start,
    row_end,
    A_ptr_start,
    A_ptr_end,
    estimated_cycle
}
```

其中：

```text
A_ptr_start = A_row_ptr[row_start]
A_ptr_end   = A_row_ptr[row_end + 1]
```

例如：

```text
PE0:
    valid = 1
    row_start = 0
    row_end = 63
    A_ptr_start = A_row_ptr[0]
    A_ptr_end = A_row_ptr[64]
    estimated_cycle = 1024

PE1:
    valid = 1
    row_start = 64
    row_end = 127
    A_ptr_start = A_row_ptr[64]
    A_ptr_end = A_row_ptr[128]
    estimated_cycle = 1008
```

PE 启动后只需要根据 `row_start` 和 `row_end` 顺序扫描 A 行。

---

# 六、Load Module 与 Global Buffer

## 1. Load Module

Load Module 根据 Decode 后的地址信息，从 DRAM 读取：

```text
A_row_ptr
A_col_idx
A_val

B_row_ptr
B_col_idx
B_val
```

并写入 Global Buffer。

由于 A/B 最大维度为 512，片上缓存压力相对较小。

---

## 2. Global Buffer

Global Buffer 用于缓存：

```text
A CSR
B CSR
b_row_nnz
row_cyc
PE task descriptor
```

建议划分为几个逻辑区域：

```text
A Buffer Region:
    A_row_ptr
    A_col_idx
    A_val

B Buffer Region:
    B_row_ptr
    B_col_idx
    B_val

Scheduler Buffer Region:
    b_row_nnz
    row_cyc
    PE_task_table
```

---

# 七、PE 本地 Buffer 设计

由于你现在设定：

```text
A/B 最大维度 512
每个 PE 的 B Buffer 保存完整 B 矩阵可以接受
```

那么 PE 内部可以设计为：

```text
A Local Buffer
B Local Buffer
Partial Row Buffer
```

---

## 1. A Buffer

每个 PE 的 A Buffer 不需要保存完整 A。

它只缓存当前 PE 负责的 A 行数据：

```text
PE0 保存 A row_start[0] ~ row_end[0]
PE1 保存 A row_start[1] ~ row_end[1]
...
```

A Buffer 内容包括：

```text
local_A_row_ptr
local_A_col_idx
local_A_val
```

也可以不重新生成 local CSR，而是直接保存全局 A 指针范围：

```text
A_ptr_start
A_ptr_end
```

PE 运行时从 Global Buffer 读取即可。

---

## 2. B Buffer

每个 PE 保存完整 B CSR：

```text
B_row_ptr
B_col_idx
B_val
```

这样每个 PE 在访问 `B(k,:)` 时，不需要争抢共享 B Buffer。

优点：

```text
1. PE 之间 B 访问互不冲突
2. 控制简单
3. 适合 512 以内的小规模矩阵
4. 可以提高并行度
```

缺点：

```text
B 被复制 N_PE 份，SRAM 开销增加
```

但在当前规模下可以接受。

---

# 八、PE 内部整体计算流程

每个 PE 拿到任务描述符：

```text
row_start
row_end
```

然后执行：

```text
for i = row_start to row_end:

    clear Partial Row Buffer

    for p = A_row_ptr[i] to A_row_ptr[i+1] - 1:

        k = A_col_idx[p]
        a = A_val[p]

        b_start = B_row_ptr[k]
        b_end   = B_row_ptr[k+1]

        for q = b_start to b_end - 1:

            j = B_col_idx[q]
            b = B_val[q]

            partial = a × b

            acc[j] += partial

    output C row i
```

---

# 九、Decompress Unit 细化

Decompress Unit 建议更准确地理解为：

```text
Sparse Decode / Stream Generation Unit
```

它不是简单解压，而是把 CSR 格式的稀疏数据展开成 MAC 阵列可以执行的乘法流。

---

## 1. Decompress Unit 输入

来自 A Buffer：

```text
A_row_ptr
A_col_idx
A_val
```

来自 B Buffer：

```text
B_row_ptr
B_col_idx
B_val
```

来自 PE Task Descriptor：

```text
row_start
row_end
```

---

## 2. Decompress Unit 内部子模块

建议拆成：

```text
1. A Row Controller
2. A Element Fetcher
3. B Row Controller
4. B Element Streamer
5. MAC Lane Dispatcher
```

---

### 2.1 A Row Controller

负责控制当前 PE 处理哪一行 A。

状态：

```text
current_row
row_end
```

执行：

```text
for current_row = row_start to row_end:
    处理当前 A 行
```

每处理新的一行时：

```text
清空 Partial Row Buffer
初始化当前行输出状态
```

---

### 2.2 A Element Fetcher

负责读取当前 A 行中的每个非零元素。

对于当前行 `i`：

```text
a_start = A_row_ptr[i]
a_end   = A_row_ptr[i+1]
```

然后逐个读取：

```text
p = a_start, a_start+1, ..., a_end-1
```

得到：

```text
k = A_col_idx[p]
a = A_val[p]
```

这个 `A(i,k)` 会广播给多个 MAC lane，与 B 第 k 行的多个非零元素相乘。

---

### 2.3 B Row Controller

对于当前 A 元素：

```text
A(i,k)
```

B Row Controller 读取：

```text
b_start = B_row_ptr[k]
b_end   = B_row_ptr[k+1]
b_len   = b_end - b_start
```

然后启动 B Element Streamer。

---

### 2.4 B Element Streamer

它负责从 B Buffer 中连续读取：

```text
B(k,j0), B(k,j1), B(k,j2), ...
```

每个 B 元素包括：

```text
j = B_col_idx[q]
b = B_val[q]
```

由于每个 PE 内有 `N_MAC` 个乘法器，所以每个周期最多取出 `N_MAC` 个 B 元素。

例如 `N_MAC = 4`：

```text
Cycle 0:
    B(k,j0), B(k,j1), B(k,j2), B(k,j3)

Cycle 1:
    B(k,j4), B(k,j5), B(k,j6), B(k,j7)

Cycle 2:
    B(k,j8), B(k,j9), idle, idle
```

---

### 2.5 MAC Lane Dispatcher

它把数据分发给各个 MAC lane。

每个 MAC lane 收到：

```text
valid
A_row_id = i
A_col_id = k
A_value = a
B_col_id = j
B_value = b
```

输出给 MUL：

```text
partial_value = A_value × B_value
target_col = j
target_row = i
```

---

## 3. Decompress Unit 输出

输出到 MUL 阵列的数据流为：

```text
lane_valid[m]
lane_a_val[m]
lane_b_val[m]
lane_col_idx[m]
lane_row_idx[m]
```

其中 `m` 是 MAC lane 编号。

---

# 十、MUL 阵列细化

MUL 阵列由 `N_MAC` 个乘法器组成。

例如：

```text
N_MAC = 4
```

则每个 PE 内有：

```text
MUL0
MUL1
MUL2
MUL3
```

每个周期最多产生 4 个 partial product。

每个 MUL 的输入：

```text
valid
a_val
b_val
col_idx j
row_idx i
```

每个 MUL 的输出：

```text
valid
partial_value = a_val × b_val
col_idx j
row_idx i
```

注意，`col_idx j` 必须随 partial product 一起传给 Aggregation Unit。
否则 Aggregation Unit 不知道这个乘积应该累加到 `C(i,j)` 的哪个位置。

---

# 十一、Aggregation Unit 细化

Aggregation Unit 是 PE 内部负责 partial products 聚合的模块。

它的核心任务是：

```text
把多个 partial product 按照列索引 j 累加到当前 C 行中
```

即：

[
acc[j] = acc[j] + partial
]

---

## 1. Aggregation Unit 输入

来自 MUL 阵列：

```text
mul_valid[m]
partial_value[m]
col_idx[m]
row_idx[m]
```

由于 PE 当前只处理一行 C，所以 `row_idx` 通常是当前行，可以不作为查找 key。

真正的聚合 key 是：

```text
col_idx j
```

---

## 2. Aggregation Unit 输出

写入 Partial Row Buffer：

```text
acc_valid[j]
acc_val[j]
```

并维护：

```text
touched_cols
```

用于记录当前 C 行有哪些列被写过。

---

## 3. Aggregation Unit 内部逻辑

对于每个有效 partial product：

```text
j = col_idx
v = partial_value
```

执行：

```text
if acc_valid[j] == 0:

    acc_valid[j] = 1
    acc_val[j] = v
    touched_cols.push(j)

else:

    acc_val[j] = acc_val[j] + v
```

这就是 PE-local sparse accumulator，也就是 SPA。

---

## 4. 同周期多个 MAC 写入的问题

因为一个周期可能有多个 MUL 输出：

```text
MUL0 → col j0
MUL1 → col j1
MUL2 → col j2
MUL3 → col j3
```

Aggregation Unit 必须处理多路写入。

有两种实现方式。

---

### 方案一：多端口 Partial Row Buffer

为 `N_MAC` 个 lane 提供多读多写端口。

优点：

```text
吞吐高
可以每周期聚合多个 partial product
```

缺点：

```text
多端口 SRAM 或寄存器阵列面积较大
```

由于最大列数 512，且 N_MAC 不大，这个方案可以考虑。

---

### 方案二：Banked Partial Row Buffer

把 Partial Row Buffer 分成多个 bank。

例如：

```text
bank_id = col_idx % N_BANK
```

不同 bank 可以并行访问。

如果同一周期多个 lane 访问同一个 bank，则发生冲突。

冲突处理方式：

```text
1. 暂停部分 lane
2. 将冲突的 partial product 放入 pending queue
3. 下一周期继续聚合
```

优点：

```text
硬件更现实
扩展性更好
```

缺点：

```text
存在 bank conflict
控制复杂一些
```

---

### 方案三：聚合输入串行化

每个周期 MUL 阵列产生多个 partial product，但 Aggregation Unit 每周期只处理一个。

优点：

```text
硬件最简单
```

缺点：

```text
Aggregation Unit 可能成为瓶颈
MUL 阵列吞吐无法充分利用
```

当前设计中如果想体现加速效果，不建议采用完全串行聚合。

---

## 5. 推荐实现

在你当前规模下：

```text
最大列数 512
N_MAC = 4
每个 PE 的规模可控
```

我建议采用：

```text
Banked Partial Row Buffer + 简单冲突队列
```

或者为了前期软仿和架构说明简化，先采用理想多端口 SPA：

```text
假设每个周期可以处理 N_MAC 个不同 col_idx 的聚合请求。
若多个 lane 访问同一列或同一 bank，则后续版本建模冲突。
```

---

# 十二、ADD 阵列细化

ADD 阵列不是简单地把 MUL 结果相加，而是负责执行：

[
acc[j] = acc[j] + partial
]

所以每个 ADD lane 对应一个 accumulator 更新操作。

---

## 1. ADD 阵列输入

每个 ADD lane 的输入包括：

```text
valid
partial_value
old_acc_value
col_idx j
is_new_col
```

其中：

```text
old_acc_value = acc_val[j]
```

如果 `acc_valid[j] == 0`，则：

```text
old_acc_value = 0
is_new_col = 1
```

---

## 2. ADD 阵列计算

对于每个 lane：

```text
new_acc_value = old_acc_value + partial_value
```

然后写回：

```text
acc_val[j] = new_acc_value
acc_valid[j] = 1
```

如果是新列：

```text
touched_cols.push(j)
```

---

## 3. ADD 阵列与 Aggregation Unit 的关系

推荐理解为：

```text
Aggregation Unit = 地址查询 + valid 判断 + 读 Partial Row Buffer + ADD 阵列 + 写回控制
```

ADD 阵列只是 Aggregation Unit 中的算术部分。

完整流程：

```text
MUL 输出 partial product
        ↓
Aggregation Unit 读取 col_idx j
        ↓
查询 acc_valid[j]
        ↓
读取 old_acc_value
        ↓
ADD lane 计算 old_acc_value + partial
        ↓
写回 Partial Row Buffer[j]
```

---

## 4. 同一周期同一列冲突

理论上可能出现：

```text
MUL0 输出 col_idx = 7
MUL1 输出 col_idx = 7
```

此时两个 partial product 都要更新：

```text
acc[7]
```

不能简单并行写回，否则会丢失一次累加。

处理方式有三种：

### 方式一：同列合并后再写

先在 Aggregation Unit 内部比较本周期的 col_idx。

如果两个 lane 的 `col_idx` 相同，则先合并：

```text
partial_sum = partial0 + partial1
```

然后只对 `acc[7]` 做一次更新。

适合 N_MAC 小的场景。

---

### 方式二：冲突 lane 延后一拍

检测到同列冲突后，只允许一个 lane 写回，其余 lane 进入 pending queue。

---

### 方式三：假设 B 同一行列索引不重复

对于 CSR 格式的 B，同一行内一般不会有重复列索引。

由于每个周期通常处理的是同一个 `B(k,:)` 的连续元素，所以同一周期内 `col_idx` 重复概率很低。

但不同 `A(i,k)` 对同一 C 行的同一列 j 会在不同时间出现，所以不会造成同周期冲突，只需要正常累加即可。

当前设计可以采用方式三作为基本假设，并在说明中补一句：

```text
由于 B 的单行 CSR 列索引通常不重复，同一周期来自同一 B 行的多个 partial product 目标列一般不同，因此同列写冲突较少；若发生同列或同 bank 冲突，可通过 pending queue 延迟处理。
```

---

# 十三、Partial Row Buffer / SPA 细化

Partial Row Buffer 保存当前 C 行的部分和。

因为最大列数为 512，所以每个 PE 可以直接维护：

```text
acc_val[512]
acc_valid[512]
```

同时维护：

```text
touched_cols
```

---

## 1. acc_val

保存当前 C 行中每个列位置的累加值：

```text
acc_val[j] = 当前 C(i,j) 的部分和
```

---

## 2. acc_valid

标记当前 C 行的第 j 列是否已经出现：

```text
acc_valid[j] = 1 表示 acc_val[j] 有效
acc_valid[j] = 0 表示当前行还没有生成 C(i,j)
```

---

## 3. touched_cols

记录当前行中出现过的列索引。

例如当前 C 行有：

```text
C(i,2)
C(i,5)
C(i,8)
```

则：

```text
touched_cols = [2, 5, 8]
```

它的作用是：

```text
1. 当前行结束后，只输出 touched_cols 对应的 acc_val
2. 清空当前行时，只清 touched_cols 对应位置
```

这样不需要每行结束后清空 512 个 acc_valid，减少开销。

---

# 十四、C CSR Writer 细化

C CSR Writer 负责把 PE 产生的每一行 C 结果转换成最终 CSR 格式。

最终输出：

```text
C_row_ptr
C_col_idx
C_val
```

---

## 1. PE 输出给 Output Buffer 的内容

每个 PE 处理完一行 C 后，输出：

```text
row_id
row_nnz
col_idx_list
val_list
```

例如：

```text
row_id = 10
row_nnz = 3
col_idx_list = [2, 5, 8]
val_list     = [1.3, 4.7, 2.1]
```

---

## 2. C CSR Writer 的主要任务

C CSR Writer 需要完成三件事：

```text
1. 收集每一行的 row_nnz
2. 生成 C_row_ptr
3. 将每一行的 col_idx 和 val 写入最终 C_col_idx / C_val
```

---

## 3. 推荐两阶段写回方案

由于 C 每行 nnz 计算前未知，最简单稳妥的是两阶段方案。

---

### 阶段一：收集 C 行结果

各 PE 把每一行的结果写入 Output Buffer。

Output Buffer 中可以保存：

```text
C_row_nnz[row_id]
C_row_col_list[row_id]
C_row_val_list[row_id]
```

因为每个 C 行只由一个 PE 生成，所以不会有多个 PE 同时写同一个 row_id 的问题。

---

### 阶段二：生成 C_row_ptr

当所有 PE 完成后，对 `C_row_nnz` 做 prefix-sum：

```text
C_row_ptr[0] = 0

for i = 0 to M - 1:
    C_row_ptr[i+1] = C_row_ptr[i] + C_row_nnz[i]
```

这样得到最终每一行 C 在 `C_col_idx` 和 `C_val` 中的写入起始位置。

---

### 阶段三：写出 C_col_idx 和 C_val

对于每一行 i：

```text
base = C_row_ptr[i]

for t = 0 to C_row_nnz[i] - 1:

    C_col_idx[base + t] = C_row_col_list[i][t]
    C_val[base + t]     = C_row_val_list[i][t]
```

最终得到标准 CSR：

```text
C_row_ptr
C_col_idx
C_val
```

---

## 4. C CSR Writer 内部模块

建议 C CSR Writer 包含：

```text
1. Row Result Collector
2. C Row NNZ Buffer
3. Prefix Sum Unit
4. CSR Address Generator
5. C Data Writer
```

---

### 4.1 Row Result Collector

从 PE 接收：

```text
row_id
row_nnz
col_idx_list
val_list
```

写入 Output Buffer。

---

### 4.2 C Row NNZ Buffer

保存：

```text
C_row_nnz[i]
```

长度最大为 512。

---

### 4.3 Prefix Sum Unit

对 `C_row_nnz` 做前缀和，生成：

```text
C_row_ptr
```

---

### 4.4 CSR Address Generator

根据：

```text
C_row_ptr[i]
```

生成每一行 C 的写入地址。

---

### 4.5 C Data Writer

将：

```text
C_col_idx
C_val
```

写入 Output Buffer 或直接通过 Store Module 写回 DRAM。

---

# 十五、Store Module

Store Module 负责把 C CSR Writer 生成的结果写回 DRAM。

写回内容：

```text
C_row_ptr
C_col_idx
C_val
```

写回顺序可以是：

```text
1. C_row_ptr
2. C_col_idx
3. C_val
```

也可以根据 DRAM burst 访问需求进行分块写回。

---

# 十六、更新后的架构模块说明总结

你可以把完整架构说明写成下面这个版本。

---

## 完整版架构说明

本设计面向稀疏矩阵乘法 SpGEMM，完成：

[
C = A \times B
]

其中 A 和 B 均采用 CSR 格式输入。Host 首先向加速器下发指令描述符，包括矩阵基地址、矩阵维度、非零元素数量、PE 数量、MAC 数量等信息。Fetch 模块从 DRAM 或控制寄存器中获取指令，Decode Unit 对指令进行解析，并分别控制 Load Module、Scheduler、PE Array 和 Store Module。

Load Module 根据 Decode 后的地址信息，从 DRAM 中读取 A 的 CSR 数据和 B 的 CSR 数据，并写入 Global Buffer。由于 B 采用 CSR 格式，Scheduler 可以直接通过 `B_row_ptr[k+1] - B_row_ptr[k]` 得到 B 第 k 行的非零元素数量 `b_row_nnz[k]`，不再需要额外的 CSC 到 row-wise 转换模块。

Scheduler 首先通过 B Row Length Generator 计算 `b_row_nnz`。随后，A Row Workload Analyzer 顺序扫描 A 的 CSR 索引。对于每个非零元素 `A(i,k)`，调度器根据 `b_row_nnz[k]` 计算其有效乘法数量以及周期权重 `ceil(b_row_nnz[k]/N_MAC)`，并累加得到每一行 A 的 `row_cyc[i]` 和全局 `TotalCycleWork`。接着，Remaining-Aware Row Partitioner 根据剩余任务量和剩余 PE 数量动态计算 `dynamic_target`，并通过最近边界判断决定每个 PE 的行切分边界。最终，Task Descriptor Generator 为每个 PE 生成 `row_start`、`row_end`、`A_ptr_start`、`A_ptr_end` 和 `estimated_cycle` 等任务描述符。

计算阶段，各 PE 根据任务描述符并行处理各自负责的连续 A 行。每个 PE 内部包含 A Buffer、B Buffer、Sparse Decode / Decompress Unit、MUL 阵列、Aggregation Unit、ADD 阵列以及 Partial Row Buffer。由于当前矩阵最大维度为 512，每个 PE 的 B Buffer 可以保存完整 B CSR 数据，从而避免 PE 之间对 B 的访问冲突。PE 处理当前 A 行时，Decompress Unit 依次读取 `A(i,k)`，并通过本地 B Buffer 中的 `B_row_ptr[k]` 和 `B_row_ptr[k+1]` 读取 B 第 k 行的所有非零元素。随后，Decompress Unit 将 `A_value`、`B_value` 和目标列索引 `j` 分发给 MUL 阵列。

MUL 阵列由多个乘法器组成，每个周期可以并行生成多个 partial product。每个 partial product 携带其目标列索引 `j`，并送入 Aggregation Unit。Aggregation Unit 根据 `j` 查询 Partial Row Buffer 中的 `acc_valid[j]` 和 `acc_val[j]`。如果该列第一次出现，则将 partial product 写入 `acc_val[j]`，并把 `j` 记录到 `touched_cols`；如果该列已经存在，则通过 ADD 阵列执行 `acc_val[j] = acc_val[j] + partial_value`。这样，当前 C 行中的所有 partial products 都在 PE 内部完成聚合。

当当前 A 行处理完成后，Partial Row Buffer 中的有效项即为当前 C 行的非零结果。PE 将 `row_id`、`row_nnz`、`C_col_idx` 和 `C_val` 发送到 Output Buffer。由于本设计采用行级顺序切分，每个 C 行只由一个 PE 负责，因此不需要跨 PE 合并同一行 C 的 partial sums。

最后，C CSR Writer 收集所有 PE 输出的行结果，首先记录每一行的 `C_row_nnz`，然后通过 Prefix Sum Unit 生成 `C_row_ptr`。之后，CSR Address Generator 根据 `C_row_ptr` 为每一行生成写入地址，并由 C Data Writer 写出最终的 `C_col_idx` 和 `C_val`。Store Module 最终将完整的 C CSR 数据写回 DRAM。

# 十八、当前版本最核心的硬件特点

可以总结成这几句话：

```text
1. A/B 均采用 CSR 输入，B 的行访问天然支持 A(i,k) × B(k,:)。
2. Scheduler 通过 B_row_ptr 直接计算 b_row_nnz，降低预处理复杂度。
3. A Row Workload Analyzer 根据 b_row_nnz 估计每一行 A 的真实计算周期。
4. Remaining-Aware Row Partitioner 使用动态剩余目标和最近边界判断进行连续行切分。
5. 每个 PE 负责连续 A 行，因此每个 C 行只在一个 PE 内完成聚合。
6. 每个 PE 本地保存完整 B CSR，减少 B 访问冲突。
7. Decompress Unit 将 CSR 稀疏数据展开为 MAC 阵列可执行的乘法流。
8. Aggregation Unit + ADD 阵列 + Partial Row Buffer 实现 PE 内部 SPA 累加。
9. C CSR Writer 负责从各 PE 行结果生成最终 C_row_ptr、C_col_idx 和 C_val。
```

这个版本已经比之前的图更完整，也更符合你现在的硬件实现设定。

同志，下面是我们这几轮讨论后形成的**新版 SpGEMM-only 稀疏矩阵加速器方案**。这版方案的核心思想是：

> **Host 端负责 A 矩阵分组和格式转换；FPGA 端只负责加载数据、广播 B、PE 并行计算、输出 dense C。当前只做 SpGEMM，不做 ADD/SUB，不追求极限性能，优先保证功能正确和 RTL 可实现。**

---

# 一、总体设计目标

当前加速器只支持：

```text
C = A × B
```

暂时不考虑：

```text
SpADD
SpSUB
CSR 格式 C 输出
复杂 scheduler
动态任务调度
复杂 global buffer
多 PE 共享 B 访问仲裁
```

新的设计目标是：

```text
1. Host 端预处理 A 和 B；
2. A 按 PE 分组；
3. B 转换成适合按行访问的格式；
4. FPGA 将完整 B 广播复制到每个 PE；
5. 每个 PE 计算自己负责的 A 行；
6. C 输出为 dense 稠密矩阵；
7. PE 计算完一行后，把这一行结果写入片上 C_dense_buffer；
8. 所有 PE 完成后，再把 C_dense_buffer 写回 DDR。
```

---

# 二、整体加速器流程

推荐顶层流程：

```text
fetch/decode
    ↓
load_descriptor
    ↓
load_B_to_all_PE
    ↓
load_A_group_to_each_PE
    ↓
start_all_PE
    ↓
each_PE_compute_own_global_rows
    ↓
completed_row_write_to_C_dense_buffer
    ↓
all_PE_done
    ↓
write_C_dense_buffer_to_DDR
    ↓
finish
```

对应顶层 FSM 可以设计为：

```text
S_IDLE
S_FETCH_DECODE
S_LOAD_DESC
S_LOAD_B_TO_ALL_PE
S_LOAD_A_GROUP_TO_PE
S_START_PE
S_WAIT_PE_DONE_AND_FILL_C_BUFFER
S_WRITE_C_DENSE_TO_DDR
S_FINISH
```

其中：

```text
S_WAIT_PE_DONE_AND_FILL_C_BUFFER
```

阶段里，每个 PE 可以边计算边把已完成的 C 行写入片上 `C_dense_buffer`。

# 三、模块划分

推荐顶层模块：

```text
core_top
 ├── fetch_decode
 ├── descriptor_loader
 ├── b_broadcast_loader
 ├── a_group_loader
 ├── pe_array
 │    ├── pe_top[0]
 │    ├── pe_top[1]
 │    ├── pe_top[2]
 │    └── ...
 ├── c_dense_buffer / c_dense_banks
 ├── c_dense_write_arbiter
 └── c_dense_ddr_writer
```

建议暂时删除或停用：

```text
sp_elementwise
ADD/SUB opcode
global CSR writer
C row_ptr / C col / C val writer
CSR prefix sum
复杂 task_loader
复杂 scheduler
复杂 pe_decompress
复杂 pe_aggregation
多 PE 共享 B buffer 仲裁
```

# 四、Host 端职责

Host 端负责做比较复杂但软件容易完成的事情：

```text
1. 读取原始 A、B 矩阵；
2. 将 A 按 PE 分组；
3. 为每个 PE 生成 A_group；
4. A_group 中每一行保留 global_row_id；
5. 将 B 转换成按行访问的 compact row-desc 格式；
6. 在 DDR 固定区域写入 A_group、B、descriptor；
7. 为 C_dense 分配固定 DDR 输出空间；
8. testbench 中生成 golden dense C 结果。
```

---

# 五、为什么 A/B 不再用固定 160 槽位

之前讨论过一种方案：

```text
每行固定 160 个非零槽位
```

后来你指出这个方案有问题：

```text
1. A 大多数行可能远小于 160 个非零，浪费大；
2. A 按 PE 分组后，每个 PE 不一定刚好 64 行；
3. B 是列稀疏，不是行稀疏，所以 B 每行可能超过 160 个非零。
```

所以最终不采用固定 160 行槽位，而采用：

```text
固定大区域基地址 + compact row-desc 内部格式
```

也就是：

```text
A/B 的大区域地址固定；
每行起始位置和 nnz 由 row_desc 指定；
C 因为输出 dense，仍然可以每行固定地址。
```

---

# 六、矩阵数据格式

赛题要求矩阵数据采用 16-bit float，所以规划为：

```text
A_val: FP16
B_val: FP16
C_val: FP16
```

索引和元数据建议：

```text
col_idx:       16-bit
global_row_id: 16-bit
row_nnz:       16-bit
start_offset:  32-bit
```

内部累加有两个选择：

```text
方案 A：acc_buffer 使用 FP16
优点：硬件简单，和输出一致
缺点：精度低，累加误差较大

方案 B：acc_buffer 使用 FP32，最后转 FP16 输出
优点：数值更合理
缺点：FP32 加法器和转换逻辑复杂
```

当前如果先追求 RTL 控制流跑通，可以先用：

```text
acc_buffer = FP16
```

后续再考虑 FP32 accumulate。

---

# 七、A_group 存储格式

A 被 host 按 PE 分组。每个 PE 有自己的 A_group。

A_group 不再使用普通 CSR，也不使用固定每行 160 槽位，而使用：

```text
A_row_desc
A_col_compact
A_val_compact
```

其中每个 `A_row_desc` 描述一行 A：

```text
A_row_desc[r] = {
    global_row_id,
    row_nnz,
    start_offset
}
```

含义：

```text
global_row_id：这一行在原始 A 矩阵里的真实行号；
row_nnz：这一行实际有多少个非零元素；
start_offset：这一行第一个非零元素在 A_col/A_val compact 数组中的位置。
```

PE 访问第 r 行第 t 个非零：

```text
desc = A_row_desc[r]

global_row_id = desc.global_row_id
a_row_nnz     = desc.row_nnz
a_start       = desc.start_offset

a_idx = a_start + t

k     = A_col_compact[a_idx]
a_val = A_val_compact[a_idx]
```

其中：

```text
k 是 A(i,k) 的列索引，也是后续访问 B 第 k 行的行索引。
```

---

# 八、A_group 容量规划

假设：

```text
MAX_DIM = 512
N_PE = 8
A 每行最大行重 30%
```

A 的最大总 nnz：

```text
ceil(512 × 0.3) = 154

A_MAX_NNZ_TOTAL = 512 × 154 = 78,848
```

为了允许 PE 之间负载不完全均衡，建议每个 PE 的容量不要只设成 64 行，而设大一些：

```text
A_ROW_SLOT_PER_PE = 128
A_NNZ_SLOT_PER_PE = 16,384
```

8 个 PE 总 A_nnz 容量：

```text
8 × 16,384 = 131,072
```

大于最大理论 A nnz：

```text
78,848
```

所以有足够余量。

每个 PE 的 A_group 容量：

```text
A_row_desc = 128 × 8 bytes = 1 KB

A_col      = 16,384 × 2 bytes = 32 KB
A_val      = 16,384 × 2 bytes = 32 KB

单 PE A_group ≈ 65 KB
```

建议每个 PE 的 A_group 区域按：

```text
A_GROUP_STRIDE = 128 KB = 0x20000
```

对齐。

---

# 九、A_group DDR 地址规划

推荐 DDR 中 A_group 固定地址：

```text
A_GROUPS_DDR_BASE = 0x0000_0000
A_GROUP_STRIDE    = 0x0002_0000   // 128 KB
```

第 p 个 PE 的 A_group 基地址：

```text
A_PE_BASE[p] = A_GROUPS_DDR_BASE + p × A_GROUP_STRIDE
```

每个 A_group 内部：

```text
A_ROW_DESC_OFFSET = 0x00000
A_COL_OFFSET      = 0x01000
A_VAL_OFFSET      = 0x09000
```

所以：

```text
PEp_A_ROW_DESC_BASE = A_PE_BASE[p] + 0x00000
PEp_A_COL_BASE      = A_PE_BASE[p] + 0x01000
PEp_A_VAL_BASE      = A_PE_BASE[p] + 0x09000
```

A 地址公式：

```text
A_row_desc_addr = PEp_A_ROW_DESC_BASE + r × 8

A_col_addr      = PEp_A_COL_BASE + a_idx × 2
A_val_addr      = PEp_A_VAL_BASE + a_idx × 2
```

# 十、B 矩阵存储格式

虽然赛题说 B 是**列稀疏**，但是 SpGEMM 计算时，PE 根据：

```text
A(i,k)
```

需要访问：

```text
B(k,:)
```

也就是 B 的第 k 行。

所以 host 应该把 B 从原始格式转换成**按行访问的 compact row-desc 格式**：

```text
B_row_desc[0 : K-1]
B_col_compact[0 : B_nnz-1]
B_val_compact[0 : B_nnz-1]
```

每个 B_row_desc：

```text
B_row_desc[k] = {
    row_nnz,
    start_offset
}
```

访问 B 第 k 行第 t 个非零：

```text
b_desc = B_row_desc[k]

b_row_nnz = b_desc.row_nnz
b_start   = b_desc.start_offset

b_idx = b_start + t

j     = B_col_compact[b_idx]
b_val = B_val_compact[b_idx]
```

注意：
B 虽然每列最多 30% 非零，但某一行可能超过 154 个非零。所以 B 不能用“每行固定 160 槽位”。

---

# 十一、B 容量规划

B 的每列最多 30% 非零：

```text
每列最多 154 个非零
```

B 总 nnz 最大：

```text
B_MAX_NNZ_TOTAL = 512 × 154 = 78,848
```

B compact 格式容量：

```text
B_row_desc = 512 × 8 bytes = 4 KB
B_col      = 78,848 × 2 bytes ≈ 154 KB
B_val      = 78,848 × 2 bytes ≈ 154 KB

B 总容量 ≈ 312 KB
```

建议 DDR 中 B 区域按：

```text
B_REGION_SIZE = 512 KB = 0x80000
```

预留。

---

# 十二、B DDR 地址规划

推荐固定 DDR 地址：

```text
B_DDR_BASE = 0x0020_0000
```

内部偏移：

```text
B_ROW_DESC_OFFSET = 0x00000
B_COL_OFFSET      = 0x01000
B_VAL_OFFSET      = 0x40000
```

所以：

```text
B_ROW_DESC_BASE = B_DDR_BASE + 0x00000
B_COL_BASE      = B_DDR_BASE + 0x01000
B_VAL_BASE      = B_DDR_BASE + 0x40000
```

访问公式：

```text
B_row_desc_addr = B_ROW_DESC_BASE + k × 8

B_col_addr      = B_COL_BASE + b_idx × 2
B_val_addr      = B_VAL_BASE + b_idx × 2
```

---

# 十三、B 广播到 PE 的含义

B 在 DDR 中只存一份。

加载阶段：

```text
b_broadcast_loader
```

从 DDR 顺序读取 B_row_desc、B_col、B_val。每读出一个数据，就同时写入所有 valid PE 的本地 B buffer。

也就是：

```text
DDR 读 B 一次
    ↓
PE0_B_buffer
PE1_B_buffer
PE2_B_buffer
...
```

而不是：

```text
DDR 读 B 给 PE0
DDR 再读 B 给 PE1
DDR 再读 B 给 PE2
...
```

这样做的原因：

```text
1. B 是所有 PE 都需要访问的；
2. 每个 PE 本地有完整 B，计算时不需要抢共享 B；
3. 避免共享 B buffer 的多端口 SRAM 和仲裁；
4. 用片上存储换控制简单。
```

缺点是片上 B buffer 占用大。

一份 B 大约 312 KB。如果复制到 8 个 PE：

```text
312 KB × 8 ≈ 2.5 MB
```

所以建议阶段性实现：

```text
第一阶段：N_PE = 1
第二阶段：N_PE = 2
第三阶段：N_PE = 4 / 8，视 FPGA SRAM/BRAM 资源决定
```

---

# 十四、C 输出格式

C 不输出 CSR，而输出 dense 稠密矩阵：

```text
C_dense[512][512]
```

矩阵数据为 FP16，所以容量：

```text
512 × 512 × 2 bytes = 524,288 bytes = 512 KB
```

每一行固定：

```text
512 × 2 bytes = 1024 bytes = 0x400
```

C 第 i 行起始地址固定：

```text
C_row_base[i] = C_BASE + i × 0x400
```

C[i][j] 地址：

```text
C_addr = C_BASE + (i × 512 + j) × 2
```

因为 512 是 2 的幂，所以 RTL 里地址很好算：

```verilog
C_addr = C_BASE + (((global_row_id << 9) + col_id) << 1);
```

---

# 十五、C DDR 地址规划

推荐：

```text
C_DDR_BASE = 0x0030_0000
C_DDR_SIZE = 0x0008_0000   // 512 KB
```

地址范围：

```text
C_dense: 0x0030_0000 ~ 0x0037_FFFF
```

注意：即使实际 N 小于 512，也建议 DDR 中仍然按 512 列 stride 存储。

例如实际 N=4，只检查：

```text
C[row][0:3]
```

但硬件仍按：

```text
C[row][0:511]
```

的行 stride 规划地址。

这样地址最简单。

---

# 十六、片上 C_dense_buffer 设计

你希望：

```text
PE 正在累计 C 的某一行时，数据在自己的 row_buffer 里；
一行累加完后，直接写进全局 C_dense_buffer；
所有 PE 都算完后，再把 C_dense_buffer 写回 DDR。
```

这个设计可以采用。

推荐结构：

```text
PE row_acc_buffer
    ↓
row_done
    ↓
c_dense_write_arbiter
    ↓
global C_dense_buffer
    ↓
c_dense_ddr_writer
    ↓
DDR
```

因为多个 PE 可能同时完成一行，所以需要一个：

```text
c_dense_write_arbiter
```

负责仲裁多个 PE 对 C_dense_buffer 的写请求。

逻辑上不会有行冲突，因为 host 保证每一行 A 只分配给一个 PE。但物理上可能有写端口冲突：

```text
PE0 完成一行，要写 C_dense_buffer
PE1 同时完成一行，也要写 C_dense_buffer
```

如果 C_dense_buffer 只有一个写端口，就必须排队。

---

# 十七、C_dense_buffer 写入接口

每个 PE 完成一行后，向 arbiter 发请求：

```text
row_write_req
global_row_id
row_buffer_data[0:511]
```

由于一次传 512 个 FP16 数据太宽，实际接口建议做成流式：

```text
row_valid
row_ready
row_global_id
row_col_idx
row_value
row_last
```

PE 输出一整行：

```text
cycle 0:  col=0,   value=row_buffer[0]
cycle 1:  col=1,   value=row_buffer[1]
...
cycle 511: col=511, value=row_buffer[511], row_last=1
```

arbiter 写入 C_dense_buffer 地址：

```text
C_sram_addr = global_row_id × 512 + row_col_idx
```

如果 SRAM 用 FP16 元素地址：

```text
C_sram_elem_addr = (global_row_id << 9) + row_col_idx
```

如果用 byte address：

```text
C_sram_byte_addr = ((global_row_id << 9) + row_col_idx) << 1
```

---

# 十八、C_dense_buffer 的实现选择

有两种方案。

## 方案 A：单个全局 C_dense_buffer + 写仲裁

```text
PE0 ┐
PE1 ├── c_dense_write_arbiter → C_dense_buffer
PE2 ┘
```

优点：

```text
逻辑简单；
C 地址就是全局 dense 地址；
最后写 DDR 很简单。
```

缺点：

```text
多个 PE 同时完成一行时需要排队；
PE 可能会等待 row_buffer 写出完成。
```

这是当前最推荐的第一版。

---

## 方案 B：C_dense_buffer 按 bank 分开

例如：

```text
C_bank0
C_bank1
C_bank2
C_bank3
```

可以按 row_id 低位分 bank：

```text
bank_id = global_row_id % NUM_BANK
```

优点：

```text
可以提高写并行度。
```

缺点：

```text
最后写 DDR 需要按全局行顺序从多个 bank 读；
控制更复杂。
```

当前先不建议做。

---

# 十九、descriptor 的作用

`descriptor` 是 host 写给硬件的一张任务配置表。

因为我们不希望 FPGA 内部做复杂 scheduler，所以 host 要告诉硬件：

```text
矩阵尺寸是多少？
哪些 PE 有效？
每个 PE 有多少行？
A_group 放在哪里？
B 放在哪里？
C 放在哪里？
```

不过因为我们已经固定了 DDR 基地址，descriptor 可以很简化。

---

# 二十、最小 descriptor / control register

如果 A/B/C 的 DDR 地址都固定，那么 descriptor 只需要：

```text
M
K
N
pe_valid_mask
PE0_row_count
PE1_row_count
...
PE7_row_count
```

建议控制寄存器：

```text
REG0: control
      bit[0] = start
      bit[1] = clear

REG1: status
      bit[0] = done
      bit[1] = busy
      bit[2] = error

REG2: M
REG3: K
REG4: N
REG5: pe_valid_mask

REG6 : PE0_row_count
REG7 : PE1_row_count
REG8 : PE2_row_count
REG9 : PE3_row_count
REG10: PE4_row_count
REG11: PE5_row_count
REG12: PE6_row_count
REG13: PE7_row_count
```

如果后续想让 DDR base 可变，再加：

```text
A_groups_base
B_base
C_base
```

但当前为了简化，可以把它们写死为参数。

---

# 二十一、总体 DDR 地址表

推荐 DDR 固定地址如下：

| 区域                   |          起始地址 |                     大小 | 说明               |
| -------------------- | ------------: | ---------------------: | ---------------- |
| A_groups             | `0x0000_0000` |   `0x0010_0000` / 1 MB | 8 个 PE，每个 128 KB |
| B_matrix_row_format  | `0x0020_0000` | `0x0008_0000` / 512 KB | B 的按行 compact 格式 |
| C_dense              | `0x0030_0000` | `0x0008_0000` / 512 KB | C dense FP16     |
| Descriptor / Control | `0x0040_0000` |   `0x0000_1000` / 4 KB | 可选任务参数           |

A_groups 内部：

```text
A_PE_BASE[p] = 0x0000_0000 + p × 0x20000

A_ROW_DESC_BASE[p] = A_PE_BASE[p] + 0x00000
A_COL_BASE[p]      = A_PE_BASE[p] + 0x01000
A_VAL_BASE[p]      = A_PE_BASE[p] + 0x09000
```

B 内部：

```text
B_ROW_DESC_BASE = 0x0020_0000
B_COL_BASE      = 0x0020_1000
B_VAL_BASE      = 0x0024_0000
```

C：

```text
C_DDR_BASE      = 0x0030_0000
```

# 二十二、PE 内部总体结构

每个 PE 建议包含：

```text
pe_top
 ├── A_row_desc_buffer
 ├── A_col_buffer
 ├── A_val_buffer
 │
 ├── B_row_desc_buffer
 ├── B_col_buffer
 ├── B_val_buffer
 │
 ├── a_row_iterator
 ├── b_row_streamer
 ├── product_task_packer / task_fifo
 ├── mac_array_4lane
 ├── product_fifo
 ├── accumulator
 ├── row_buffer
 └── row_writeback_to_C_dense_buffer
```

PE 总体数据流：

```text
A_row_desc
    ↓
a_row_iterator
    ↓
读取 A(i,k)
    ↓
b_row_streamer 读取 B(k,:)
    ↓
product task stream
    ↓
task_packer 填满 4 个 MAC
    ↓
4-lane MAC
    ↓
product_fifo
    ↓
serial accumulator 更新 acc[j]
    ↓
row_buffer / acc_buffer
    ↓
一行完成后写 C_dense_buffer
```

---

# 二十三、PE 的计算流程

对每个 PE：

```text
for each assigned A row:

    读取 A_row_desc:
        global_row_id
        a_row_nnz
        a_start_offset

    清空 acc_buffer[0:511]

    for each A nonzero in this row:

        k     = A_col[a_idx]
        a_val = A_val[a_idx]

        读取 B_row_desc[k]:
            b_row_nnz
            b_start_offset

        for each B nonzero in B row k:

            j     = B_col[b_idx]
            b_val = B_val[b_idx]

            生成 task:
                col_id = j
                a_val
                b_val

    task 送入 4-MAC array

    MAC 输出 product:
        product = a_val × b_val

    accumulator:
        acc[col_id] += product

    当前 A 行所有 product 处理完后:
        row_buffer[j] = acc[j]

    把 row_buffer 写入全局 C_dense_buffer 的 global_row_id 行
```

---

# 二十四、为什么需要 product_task_packer

你提出的问题是：

```text
PE 有 4 个 MAC。
A 当前行的一个非零 A(i,k0) 对应 B(k0,:)。
如果 B(k0,:) 有 6 个非零：

cycle 0 可以填 4 个 MAC；
cycle 1 只剩 2 个任务。

这时希望用 A 当前行下一个非零 A(i,k1) 对应的 B(k1,:) 的前 2 个任务填补空闲 MAC。
```

这个想法是对的。

实现方法是：

```text
不要让 MAC 直接跟 B 行读取状态机绑定；
而是先把所有乘法操作转成连续 task stream；
再由 task_packer 每周期取最多 4 个 task 喂给 MAC。
```

例如任务流：

```text
A(i,k0) × B(k0,:) 产生 6 个 task
A(i,k1) × B(k1,:) 产生 5 个 task
```

总 task stream：

```text
T0 T1 T2 T3 T4 T5 T6 T7 T8 T9 T10
```

4 个 MAC 分配：

```text
cycle 0: T0  T1  T2  T3
cycle 1: T4  T5  T6  T7
cycle 2: T8  T9  T10 invalid
```

这里 `cycle 1` 的 `T6/T7` 就来自下一个 A 非零对应的 B 行。

---

# 二十五、task_fifo 设计

推荐在 B row streamer 和 MAC array 之间放一个 task FIFO：

```text
b_row_streamer → task_fifo → 4-MAC array
```

每个 task 包含：

```text
task_col_id
task_a_val
task_b_val
```

即：

```text
task = {
    col_id,
    a_val,
    b_val
}
```

task_fifo 的作用：

```text
1. 把不规则 B 行访问转成连续 task stream；
2. 尽量让 4 个 MAC 每周期都有任务；
3. 允许不同 A(i,k) 对应的 B 行任务连续打包；
4. 隔离 B 读取和 MAC 消费速率差异。
```

---

# 二十六、MAC array 设计

4-lane MAC 输入：

```text
mac_valid[3:0]
mac_col[0:3]
mac_a_val[0:3]
mac_b_val[0:3]
```

MAC 做：

```text
product[lane] = mac_a_val[lane] × mac_b_val[lane]
```

输出送入 product_fifo：

```text
product_valid
product_col
product_val
```

---

# 二十七、为什么 MAC 后面还需要 product_fifo

因为 4 个 MAC 一周期最多产生 4 个 product。

但 accumulator 如果是普通 SRAM / regfile，很难一周期支持 4 个随机写：

```text
acc[col0] += product0
acc[col1] += product1
acc[col2] += product2
acc[col3] += product3
```

还可能出现两个 lane 写同一个 col：

```text
lane0: acc[5] += p0
lane2: acc[5] += p2
```

所以第一版建议：

```text
4 MAC 输出 → product_fifo → 串行 accumulator
```

也就是 accumulator 每周期处理 1 个 product：

```text
product = product_fifo.pop()
acc[product_col] += product_val
```

优点：

```text
1. 正确性容易保证；
2. acc_buffer 只需要 1R1W；
3. 不需要处理多写端口冲突；
4. 后续可以再优化成多 bank accumulator。
```

缺点：

```text
aggregation 可能成为瓶颈，4 MAC 不能完全发挥性能。
```

当前阶段可以接受，因为目标是先跑通功能。

---

# 二十八、PE 内部 FSM

推荐 PE FSM：

```text
PE_IDLE
PE_LOAD_ROW_DESC
PE_CLEAR_ACC
PE_GEN_TASKS
PE_WAIT_TASK_FIFO_EMPTY
PE_DRAIN_PRODUCT_FIFO
PE_WRITE_ROW_TO_C_BUFFER
PE_NEXT_ROW
PE_DONE
```

状态含义：

```text
PE_IDLE:
    等待 start pulse。

PE_LOAD_ROW_DESC:
    读取当前 A 行的 global_row_id、row_nnz、start_offset。

PE_CLEAR_ACC:
    清空 acc_buffer[0:511]。

PE_GEN_TASKS:
    遍历当前 A 行所有 A(i,k)，读取对应 B(k,:) 并生成 task。

PE_WAIT_TASK_FIFO_EMPTY:
    当前行所有 task 已生成，等待 task_fifo 被 MAC 消费完。

PE_DRAIN_PRODUCT_FIFO:
    等待 MAC 结果全部进入 accumulator 并完成累加。

PE_WRITE_ROW_TO_C_BUFFER:
    将 acc_buffer / row_buffer 写入全局 C_dense_buffer 的 global_row_id 行。

PE_NEXT_ROW:
    处理下一行。

PE_DONE:
    当前 PE 所有 row_count 行处理完成。
```

---

# 二十九、不能跨 A 行填 MAC

task_packer 可以跨同一 A 行里的多个 A 非零元素填满 MAC。

也就是可以混合：

```text
A(i,k0) 对应的 B(k0,:)
A(i,k1) 对应的 B(k1,:)
```

因为它们都贡献到同一个 C 行：

```text
C(i,:)
```

但是当前阶段不要跨 A 的不同行混合任务。

也就是不要一个 cycle 里：

```text
lane0/lane1 计算 C(i,:)
lane2/lane3 计算 C(i+1,:)
```

除非 accumulator 支持多行同时累加，否则会把 acc_buffer 设计复杂化。

当前规则：

```text
一个 PE 同一时间只处理一个 C 行；
task_packer 只在同一个 C 行内部填满 4 个 MAC；
当前 C 行处理完后，再处理下一行。
```

---

# 三十、顶层启动和完成信号

PE start 必须是一拍 pulse：

```text
pe_start_pulse
```

不能是整个 compute 状态的高电平。

PE done：

```text
pe_done[p] = 当前 PE 所有 row_count 行处理完成
```

顶层 all done：

```text
pe_all_done = &(pe_done | ~pe_valid)
```

必须 mask 掉 invalid PE。

# 三十一、推荐实现阶段

## 第一阶段：单 PE / 单 MAC / dense C

目标：

```text
N_PE = 1
N_MAC = 1
2×2 / 4×4 / 8×8 小矩阵
A_group compact row-desc
B compact row-desc
C dense FP16 输出
```

先跑通：

```text
A row iterator → B row streamer → accumulator → C_dense_buffer → DDR
```

---

## 第二阶段：单 PE / 4 MAC / task_fifo

目标：

```text
N_PE = 1
N_MAC = 4
加入 task_fifo
加入 product_fifo
验证 B 行长度不是 4 倍数时可以跨下一个 A(i,k) 填补 MAC lane
```

---

## 第三阶段：多 PE / B 广播

目标：

```text
N_PE = 2
B 广播到 PE0 / PE1
A_group 分别加载到 PE0 / PE1
两个 PE 独立计算
C_dense_write_arbiter 写全局 C_dense_buffer
```

---

## 第四阶段：更多 PE

目标：

```text
N_PE = 4 / 8
检查 SRAM/BRAM 资源
检查 B buffer 复制是否可接受
优化 c_dense_buffer 写仲裁
```

# 三十二、testbench / gen_data.py 需要同步修改

testbench 需要按新格式生成数据：

```text
1. 只生成 SpGEMM；
2. A 按 PE 分组；
3. 每个 PE 生成：
   - A_row_desc
   - A_col_compact
   - A_val_compact
4. B 转换为按行 compact row-desc 格式：
   - B_row_desc
   - B_col_compact
   - B_val_compact
5. C golden 使用 dense 格式；
6. 检查 DDR 中 C_dense 是否正确；
7. 不再检查 C_row_ptr / C_col / C_val。
```

小矩阵时也可以仍然按最大 stride 存 C：

```text
C_addr = C_BASE + (row × 512 + col) × 2
```

只检查实际：

```text
row < M
col < N
```

# 三十三、最终方案一句话总结

新版加速器采用 **SpGEMM-only 简化架构**：Host 端将 A 按 PE 分组并生成带 `global_row_id / row_nnz / start_offset` 的 compact A_group，同时将列稀疏 B 转换为按行访问的 compact row-desc 格式；FPGA 端固定地址加载 A_group 和 B，将完整 B 广播复制到每个 PE；PE 内部通过 `A row iterator → B row streamer → task_fifo → 4-MAC → product_fifo → serial accumulator` 生成 dense C 行；每行完成后写入全局 `C_dense_buffer`，所有 PE 完成后统一将 FP16 dense C 写回 DDR。

---

我建议后续让 agent 改 RTL 时，不要继续沿用旧的 `CSR writer / global buffer / ADD-SUB` 架构，而是按这版重新拆模块。最先实现的闭环应该是：

```text
N_PE = 1
N_MAC = 1
compact A/B input
dense C output
DDR 写回正确
```

这个闭环通了以后，再加 `4-MAC task_fifo` 和多 PE。

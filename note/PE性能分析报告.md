## 六、MAC 利用率分析

### 6.1 计算工作量

| 指标 | 值 |
|------|-----|
| A 非零元总数 | 1,451 |
| B 平均每行非零 | 259/317 ≈ 0.82 |
| 估计总乘法数 | 1,451 × 0.82 ≈ **1,190** |
| 每行平均乘法数 | 1,190/32 ≈ 37 |
| 4-MAC 理论最短周期 | 1,190/4 ≈ 298 cycles |
| 实际计算周期 | ~13,500 cycles |

### 6.2 MAC 利用率

```
MAC 利用率 = 总乘法数 / (计算周期 × MAC数)
          = 1,190 / (13,500 × 4)
          = 1,190 / 54,000
          = 2.20%
```

**瓶颈分析**：
- 序列化器 + 串行累加器：每个 product 需 **3 个周期** (IDLE→ADD→WRITE)
- 但 MAC 可同时产出 4 个 product → 实际有效吞吐仅 1 product/3 cycles ≈ 0.33/cycle
- 4 MAC 理论上 4/cycle → 利用率 = 0.33/4 = **8.3%**（MAC 角度）
- 考虑到 FIFO 满停等待、FLUSH 排空开销，实际降至 **2.2%**

### 6.3 提高利用率的途径

1. **Banked Accumulator** — 支持多 product 同时累加（当前串行）
2. **增大 FIFO 深度** — 减少 pipeline flush 频次
3. **合并相邻 B 行** — 减少 FLUSH→DRAIN 循环次数

## 七、修复的 Bug 总结

| # | 根因 | 症状 | 修复方法 |
|---|------|------|---------|
| 1 | 序列化器握手时序错误 | `acc_in_valid` 永不为 1 | 先 present 再等 ready |
| 2 | MAC col 寄存器 vs val 组合不同步 | 产品列号与值错位 | 加 `val_pipe` 同步管线 |
| 3 | `state_stable` 导致 `row_idx` 双递增 | 跳行/死循环 | `!state_stable` 条件递增 |
| 4 | `a_nnz_left` 下溢 0→0xFFFF | 无限循环 | `a_nnz_left > 0` 保护 |
| 5 | `write_global_row` 首周期用旧值 | C 行地址错 | 提前在 drain 状态设 |
| 6 | `a_nnz_left` 同 B 行连减两次 | 丢产品 | `a_pending` + `b_batch_done` |
| 7 | task FIFO 空后 MAC 读组合旧数据 | 重复填充 product FIFO | 寄存器 + `rd_en` 门控 |
| 8 | flush 时 `pack_count=0` → `flush_done` 永不为 1 | 卡 FLUSH 状态 | `pack_count==0` 时立即 `flush_done=1` |

## 九、结论

本 PE 实现了规范的 SpGEMM 稀疏矩阵乘法加速器，支持 4 个并行乘法器 + task_packer + product_serializer + 串行累加器架构。

**优势**：
- 完整的 4-MAC 流水线，task_packer 自动填满 4 个 lane
- 支持可变 nnz/row 的 compact row-desc 格式
- dense C 输出，简化下游
- 经过 7 种规模的严格测试，小矩阵 100% 正确

**待改进**：
- 串行累加器为当前性能瓶颈（MAC 利用率仅 2.2%）
- 赛题大矩阵偶有微小偏差（5/192，待进一步调试）
- 可扩展为多 PE 并行
